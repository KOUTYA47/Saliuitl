## 概要

このドキュメントは「復元画像の可視化」から「原因分析」までの一連のスライド構成を詳細に記述する。

**対象スライド**: Slide 9〜12（4枚構成）
**ストーリーライン**: 疑問 → 実装 → 発見 → 分析 → 検証 → 結論

## スライド構成案

#### 目的
- なぜ画像出力が必要だったかを説明
- 実装の経緯を簡潔に伝える

```
┌─────────────────────────────────────────────────────────┐
│  【タイトル】復元画像の可視化                              │
├─────────────────────────────────────────────────────────┤
│                                                         │

│    - 数値結果（RR=53.8%）だけでは不十分                   │
│    - 「復元成功」とは具体的にどういう状態？                │
│    - パッチは本当に除去されているのか？                    │
│                                                         │
│  ■ 実装の試行錯誤                        ┌────────────┐ │

│       復元ロジックが複雑すぎて再現不可    │  テキスト   │ │
│    ② saliuitl.py本体に追加 → ✅        │  のみ      │ │

```

#### 使用する図
- **なし**（テキスト中心、次スライドへの導入）

#### 話すポイント
- 「数値だけでは本当に復元できているか分からなかった」
- 「復元ロジックが複雑で、別スクリプトでは再現できなかった」
- 「本体に機能を追加して、ようやく画像を確認できるようになった」

### Slide 10: 衝撃の発見（問題提起）

#### 目的
- 実際の出力画像を見せて「問題」を視覚的に提示
- 「Success」と視覚的品質のギャップを強調

#### 内容構成

```
┌─────────────────────────────────────────────────────────┐
│  【タイトル】問題発見：パッチが残っている                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  [6パネル図: voc_1p_000006.png - Success例]      │   │
│  │  Clean(3) → Attacked(3) → Recovered(8)          │   │
│  │  「Success」判定だがパッチは残存                   │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ■ 発見した矛盾                                        │
│    | 評価結果 | 視覚的状態 |                            │
│    |----------|------------|                           │
│    | Success  | パッチ残存 |  ← これがおかしい          │
│    | Failed   | パッチ残存 |                            │
│                                                         │
│  ■ 重要な気づき                                        │
│    Recovery Rate の「成功」≠ パッチの視覚的除去          │
│    → 評価指標と期待する結果にズレがある                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 使用する図
- **メイン**: `experiments/exp_20260202_viz_improved/figures/voc_1p_000006.png`
  - Success判定だがパッチが残存している例
  - 6パネル全体を見せる（検出数の変化、マスクの状態）

#### 代替/補助図
- `experiments/exp_20260202_viz_improved/figures/voc_1p_000001.png`（Failed例との比較用）

#### 話すポイント
- 「出力してみて驚いた。Successと判定されているのにパッチが残っている」
- 「よく見ると、マスク（右下）がパッチ全体をカバーしていない」
- 「これは何かおかしいのではないか、と原因分析を開始した」

### Slide 11: 原因分析（技術的説明）

#### 目的
- なぜパッチが除去されないのかを技術的に説明
- 3つの主要因を明確に提示

#### 内容構成

```
┌─────────────────────────────────────────────────────────┐
│  【タイトル】原因分析：なぜパッチが残るのか               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────────┐  ■ 原因1: 解像度ミスマッチ（主因）    │
│  │ [因果連鎖図]  │    特徴マップ: 26×26                 │
│  │              │    入力画像:  416×416                │
│  │  低解像度     │    → 1ピクセル誤差 = 16ピクセル誤差   │
│  │     ↓        │                                      │
│  │  粗いマスク   │  ■ 原因2: マスク生成の不正確さ        │
│  │     ↓        │    DBSCANによるマスクが断片的         │
│  │  パッチ残存   │    パッチ全体をカバーできない          │
│  │              │                                      │
│  └──────────────┘  ■ 原因3: Biharmonicの限界           │
│                      曲率最小化 → 滑らか = ぼやける      │
│  [failure_analysis_summary.png の因果関係パネル]        │
│                                                         │
│  ■ 根本的な問題                                        │
│    評価指標: 検出回復（IoU≥0.5）                        │
│    期待結果: 視覚的パッチ除去                           │
│    → 最適化目標と期待にズレがある                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 使用する図
- **メイン**: `docs/slides_material/failure_analysis_summary.png`（因果関係パネル部分）
- **または**: 自作の因果連鎖図（シンプルなフロー図）

#### 必要な新規図（作成推奨）
```
┌─────────────────────────────────────────┐
│         解像度ミスマッチの図解           │
├─────────────────────────────────────────┤
│  特徴マップ (26×26)    入力画像 (416×416) │
│  ┌──┬──┬──┐          ┌────────────────┐ │
│  │  │  │  │          │                │ │
│  ├──┼──┼──┤   ×16    │   パッチ       │ │
│  │  │██│  │  ───→    │   (実際位置)   │ │
│  ├──┼──┼──┤          │                │ │
│  │  │  │  │          │ ████ (マスク)  │ │
│  └──┴──┴──┘          └────────────────┘ │
│  1セル = 16×16ピクセル                   │
│  → マスクがずれる                        │
└─────────────────────────────────────────┘
```

#### 話すポイント
- 「最大の原因は解像度のミスマッチ」
- 「26×26の特徴マップから416×416のマスクを生成するので、精度が出ない」
- 「さらにbiharmonicは大きな領域をぼやけさせる特性がある」

```
┌─────────────────────────────────────────────────────────┐
│  【タイトル】検証実験：仮説の確認                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ■ 実験1: Oracle Inpainting Test                       │
│    「正確なマスクがあれば除去できるのか？」               │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  [oracle_000001.png]                             │   │
│  │  Clean → Attacked → Oracle Inpainted            │   │
│  │  差分マスク(5.3%) → パッチ除去成功（ただしぼやける）│   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ■ 実験2: Grad-CAM分析                                 │
│    「パッチは何をしているのか？」                        │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  [gradcam_000001.png]                            │   │
│  │  Clean Attention → Attacked Attention            │   │
│  │  人物 → パッチに注意が移動（Hijacking）           │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ■ 結論                                                │
│    ✓ 正確なマスクがあればパッチ除去は可能                │
│    ✓ 現システムのボトルネックはマスク精度                │
│    △ ただしbiharmonicは結果がぼやける                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 使用する図
- **上段**: `experiments/exp_20260202_visual_recovery/oracle_test/voc/oracle_000001.png`

  - パッチが除去されているがぼやけていることを示す

- **下段**: `experiments/exp_20260202_visual_recovery/gradcam/voc/gradcam_000001.png`
  - Attention Hijackingの可視化

#### 話すポイント
- 「Oracle Testで、正確なマスクがあればパッチ除去できることを確認」
- 「Grad-CAMで、パッチがモデルの注意をハイジャックしていることを発見」
- 「つまり、問題はマスク生成の精度にある」

## キーメッセージ

1. **数値だけでは不十分** → 画像を見ることで初めて問題が分かった
2. **「成功」の定義に注意** → 評価指標と視覚的品質は別物
3. **根本原因は解像度** → 特徴マップの粗さがマスク精度を制限
4. **マスク精度が鍵** → Oracle Testで正確なマスクなら除去可能と確認
5. **手法の限界を発見** → これはバグではなく、論文手法自体の制約

## 補足：ストーリーの一貫性

### Before（従来の説明）
「Recovery Rateを再現しました」→ 数値の報告で終わり

### After（この構成）
「Recovery Rateを再現したが、画像を見たら問題を発見した」
→ 「なぜか調べた」
→ 「解像度の問題だと分かった」
→ 「実験で確認した」
→ 「手法の限界を理解した」

**聴衆へのメッセージ**: 単なる再現ではなく、深く分析して新しい知見を得た
